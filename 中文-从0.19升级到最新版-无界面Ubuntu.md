# RAGFlow 从 v0.19 升级到最新版稳定版：无界面 Ubuntu 傻瓜版

最后确认日期：2026-06-04

生产建议升级到稳定版：`v0.25.6`

不要用 `nightly`，它是预发布/测试性质，不适合正式环境。

## 先说结论

最安全做法不是在旧服务器上直接覆盖升级，而是：

1. 旧 v0.19 服务器先完整备份。
2. 新 Ubuntu 服务器放新版升级包。
3. 把旧数据恢复到新版升级包里。
4. 新服务器先跑起来测试。
5. 确认旧账号、知识库、文档、聊天助手、团队权限都正常后，再把域名或反向代理切过去。

这样就算新版出问题，旧服务器还在，不会影响生产。

## 这次升级会保留什么

RAGFlow 的数据不只在数据库里，所以必须一起保留：

- `mysql_data`：用户、知识库、助手、聊天记录、模型配置。
- `minio_data`：上传过的原始文件、解析文件。
- `esdata01`：搜索和向量索引。
- `redis_data`：缓存和运行状态。
- `.env`：密码、端口、配置。
- `custom_server/config`：你以前自定义服务配置。
- 新版定制前端和新版定制后端补丁。

只拷 MySQL 不够。只拷数据库，文档文件和索引可能对不上。

## 你的自定义逻辑怎么保留

这次不是直接用官方后端镜像，而是：

```bash
官方 infiniflow/ragflow:v0.25.6
+ 你的后端权限补丁
= ragflow:v0.25.6-custom
```

保留的旧逻辑包括：

- `user.is_create_kownledge`：谁可以创建知识库/助手。
- `dialog.permission`：助手是自己可见 `me`，还是团队可见 `team`。
- 团队成员只能看 `team` 助手。
- 共享助手下，每个人的会话互相隔离，不会串。
- 新版定制前端继续使用你的登录页、页面样式、团队用户页面等。

## 千万不要做的事

不要执行：

```bash
docker compose down -v
docker-compose down -v
```

`-v` 可能删数据卷。

也不要只改 `.env` 里的 `RAGFLOW_IMAGE` 来升级。你有自定义前端和后端权限逻辑，必须用这个升级包。

## 第一步：在旧 v0.19 服务器上确认路径

登录旧 Ubuntu 服务器：

```bash
ssh 用户名@旧服务器IP
```

找到旧 RAGFlow 目录。常见是：

```bash
cd /home/xsuper/app/ragflow
```

如果不是这个路径，用下面命令找：

```bash
find / -name docker-compose.yml 2>/dev/null | grep ragflow
```

进入旧目录后看状态：

```bash
docker compose ps
docker compose config > /tmp/ragflow-v019-compose.txt
```

看数据目录是不是这些：

```bash
ls -lh mysql_data minio_data esdata01 redis_data
```

如果这几个目录存在，说明是目录挂载，最容易迁移。

## 第二步：旧服务器做完整备份

建议维护时间做，先停旧服务，备份最干净。

假设旧目录是 `/home/xsuper/app/ragflow`：

```bash
cd /home/xsuper/app/ragflow
docker compose down
mkdir -p /home/xsuper/app/ragflow-backups
tar -czf /home/xsuper/app/ragflow-backups/ragflow-v019-full.tgz \
  mysql_data minio_data esdata01 redis_data \
  .env docker-compose.yml docker-compose-base.yml \
  service_conf.yaml.template nginx custom_server
docker compose up -d
```

备份完旧系统会重新启动。

如果 `tar` 提示某个目录不存在，不要急，先看你的旧目录里实际叫什么，再调整命令。

## 第三步：把升级包传到新 Ubuntu 服务器

新服务器建议目录：

```bash
/opt/ragflow/ragflow-v0.25.6-upgrade
```

先在新服务器准备目录：

```bash
ssh 用户名@新服务器IP
sudo mkdir -p /opt/ragflow
sudo chown -R "$USER:$USER" /opt/ragflow
```

从你现在的升级包传过去。可以用 `scp` 或 `rsync`：

```bash
rsync -a ragflow-v0.25.6-upgrade/ 用户名@新服务器IP:/opt/ragflow/ragflow-v0.25.6-upgrade/
```

再把旧服务器备份包也传到新服务器：

```bash
scp /home/xsuper/app/ragflow-backups/ragflow-v019-full.tgz 用户名@新服务器IP:/opt/ragflow/
```

## 第四步：新服务器恢复旧数据

登录新服务器：

```bash
ssh 用户名@新服务器IP
cd /opt/ragflow/ragflow-v0.25.6-upgrade
```

处理一下脚本换行和权限：

```bash
sed -i 's/\r$//' *.sh scripts/*.sh entrypoint.sh launch_backend_service.sh migration.sh
chmod +x *.sh scripts/*.sh entrypoint.sh launch_backend_service.sh migration.sh
```

解压旧数据：

```bash
mkdir -p /tmp/ragflow-v019-restore
tar -xzf /opt/ragflow/ragflow-v019-full.tgz -C /tmp/ragflow-v019-restore
```

把旧数据复制到新版目录：

```bash
rsync -a /tmp/ragflow-v019-restore/mysql_data/ ./mysql_data/
rsync -a /tmp/ragflow-v019-restore/minio_data/ ./minio_data/
rsync -a /tmp/ragflow-v019-restore/esdata01/ ./esdata01/
rsync -a /tmp/ragflow-v019-restore/redis_data/ ./redis_data/
rsync -a /tmp/ragflow-v019-restore/custom_server/config/ ./custom_server/config/
```

注意：不要用旧的 `docker-compose.yml` 覆盖新版升级包里的 `docker-compose.yml`。

`.env` 要人工对比，把旧密码、端口、模型配置搬到新版 `.env`，不要整个覆盖。

## 第五步：构建你的新版自定义后端镜像

在新服务器执行：

```bash
cd /opt/ragflow/ragflow-v0.25.6-upgrade
./build-custom-backend-image.sh
docker image inspect ragflow:v0.25.6-custom >/dev/null
docker compose config
```

如果服务器需要先拉官方镜像：

```bash
PULL_BASE=1 ./build-custom-backend-image.sh
```

如果 Docker Hub 拉取超时，可以改用这次在 246 服务器实测可用的华为云代理镜像：

```bash
BASE_IMAGE=swr.cn-north-4.myhuaweicloud.com/ddn-k8s/docker.io/infiniflow/ragflow:v0.25.6 \
PULL_BASE=1 \
./build-custom-backend-image.sh
```

如果服务器不能联网，需要先在能联网的机器上 `docker save`，传到新服务器后 `docker load`。

## 第六步：启动新版测试

```bash
cd /opt/ragflow/ragflow-v0.25.6-upgrade
docker compose up -d
docker compose ps
docker compose logs -f ragflow-cpu
```

第一次启动会自动做数据库迁移，会补这些字段：

- `user.is_create_kownledge`
- `dialog.permission`
- `conversation.user_id`

这一步正常，不是丢数据。

## 第七步：检查数据库字段有没有补上

```bash
cd /opt/ragflow/ragflow-v0.25.6-upgrade
set -a
. ./.env
set +a
docker compose exec mysql mysql -uroot -p"$MYSQL_PASSWORD" rag_flow -e "SHOW COLUMNS FROM user LIKE 'is_create_kownledge';"
docker compose exec mysql mysql -uroot -p"$MYSQL_PASSWORD" rag_flow -e "SHOW COLUMNS FROM dialog LIKE 'permission';"
docker compose exec mysql mysql -uroot -p"$MYSQL_PASSWORD" rag_flow -e "SHOW COLUMNS FROM conversation LIKE 'user_id';"
```

如果三条都有结果，说明核心后端迁移成功。

## 第八步：你要重点点开测试这些

浏览器访问新服务器地址，然后检查：

- 老账号能不能登录。
- 旧知识库还在不在。
- 旧文档能不能打开、预览、下载。
- 聊天能不能检索旧文档内容。
- 登录页和你改过的页面是不是还在。
- 团队用户里 `is_create_kownledge` 能不能显示/保存。
- 没权限的人是否不能创建知识库/助手。
- 助手设置里 `me/team` 能不能保存。
- 团队成员只能看到 `team` 助手。
- 共享助手下，A 用户看不到 B 用户的会话。

只要这些过了，就说明旧数据基本能在新版上用。

## 第九步：正式切换

最推荐的正式切换方式：

旧服务器先别动，新服务器测试通过后，把域名或 nginx 反向代理指到新服务器。

如果必须在同一台机器切换，再用升级包里的脚本：

```bash
cd /opt/ragflow/ragflow-v0.25.6-upgrade
APP_ROOT=/opt/ragflow \
OLD_DIR=/opt/ragflow/ragflow \
NEW_DIR=/opt/ragflow/ragflow-v0.25.6-upgrade \
STOP_FOR_DATA_BACKUP=1 \
./scripts/backup-current-ragflow.sh
```

然后：

```bash
APP_ROOT=/opt/ragflow \
OLD_DIR=/opt/ragflow/ragflow \
NEW_DIR=/opt/ragflow/ragflow-v0.25.6-upgrade \
I_HAVE_BACKUP=yes \
./scripts/cutover-to-v0.25.6.sh
```

如果你的旧目录不是 `/opt/ragflow/ragflow`，要把 `OLD_DIR` 改成真实路径。

## 第十步：回滚怎么做

最安全回滚：

旧服务器保持不动。新版不行，就把域名/反代切回旧服务器。

如果你是在同一台机器切换，回滚前注意：新版启动后可能已经改过数据库结构，不要直接拿新版改过的数据给 v0.19 用。要恢复 v0.19 的备份。

## 你真正需要记住的三句话

1. 先完整备份，再升级。
2. 新服务器用旧数据副本测试，不要直接动旧生产。
3. MySQL、MinIO、ES 索引都要保留，只保留数据库不够。
